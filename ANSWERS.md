#### Вопрос 1
В `BaseTest` используется `implicitlyWait(10s)`. Ваш коллега добавил явный `WebDriverWait(driver, 15s)` в одном из
тестов.
При запуске тесты стали нестабильными - иногда падают, иногда нет, без видимой причины.
Объясните возможную механику происходящего и как бы вы исправили проблему.

#### Ответ 1
Видимо проблема возникает из-за смешивания неявного и явного ожиданий. Неявное ожидание 10 сек применяется к каждому
вызову
findElement() внутри явного ожидания 15 сек. Из-за этого таймауты могут суммироваться до 25 сек, тесты становятся flaky. 
Когда элемент появляется ровно на границе таймаута (например, на 10-й секунде),
иногда неявное ожидание успевает его найти, а иногда нет - отсюда нестабильность. Также возможна проблема
StaleElementReferenceException, когда элемент успевает обновиться между проверками.
Для исрпавления проблемы нужно удалить неявное ожидание из BaseTest и использовать только явные ожидания
WebDriverWait с нужными условиями (visibilityOfElementLocated, elementToBeClickable и т.д.).

**Проблема:** Смешивание неявного (implicitlyWait) и явного (WebDriverWait) ожиданий.

**Механика:**
- Неявное ожидание применяется при КАЖДОМ вызове `findElement()`
- Явное ожидание внутри себя периодически вызывает `findElement()` для проверки условия
- В итоге каждый цикл проверки в явном ожидании длится не 500 мс (типичный poll interval), а 10 секунд (из-за неявного)

**Почему тесты становятся flaky:**
Допустим, элемент появляется на 11-й секунде:
- В одном запуске: явное ожидание сделало 2 проверки (на 0с и на 10с), элемент не нашли → упало
- В другом запуске: элемент появился чуть раньше, на 9.5с, неявное ожидание его нашло → тест прошел

**StaleElementReferenceException возникает потому, что:**
1. Элемент найден неявным ожиданием на 10-й секунде
2. Явное ожидание пытается с ним работать
3. В этот момент DOM обновляется (AJAX, перерисовка)
4. Элемент становится stale, тест падает

**Исправление:**
Удалить `implicitlyWait` полностью. Использовать только явные ожидания с конкретными условиями:
- `visibilityOfElementLocated` — для проверки появления
- `invisibilityOfElementLocated` — для проверки исчезновения
- `elementToBeClickable` — для кликабельных элементов


#### Вопрос 2
Тест-сьют из 200 UI тестов стабильно проходит локально,
но в CI падает ~15% тестов с разными ошибками: `StaleElementReferenceException`, `NoSuchElementException`,
и иногда - неожиданный `401` от API. Тесты перезапускаются и проходят. Какие 3 наиболее вероятные причины нестабильности
вы бы проверили в первую очередь, и как бы вы их диагностировали?

#### Ответ 2
Возможные причины:
-Гонка между тестами (в CI тесты часто запускаются параллельно, а локально последовательно. Один тест удаляет данные,
которые нужны другому).
Для проверки запустить в CI последовательно.
Для решения проблемы сделать тесты независимыми, использовать уникальные данные.

-Медленная загрузка в CI. (к примеру, локально элемент появляется за 2 секунды, в CI за 10, а тест ждёт 5.)
Можно попробовать увеличить таймауты явных ожиданий до 30 секунд.

-Токен протухает, в CI тесты идут дольше, чем живёт токен. Минут через 15-20 начинает сыпаться 401.
Обновлять токен перед каждым тестом или перезапрашивать при 401.

**Топ-3 причины в порядке приоритета:**

### 1. Разная скорость выполнения (тайминги)

**Почему:** Локально всё быстро, в CI — медленнее (общие ресурсы, нагруженная БД, сетевые задержки).

**Диагностика:**
- Добавить логирование времени выполнения каждого теста в CI
- Сравнить с локальными прогонами
- Увеличить таймауты до 30 секунд и посмотреть, упадут ли тесты

**Решение:** Использовать динамические ожидания (WebDriverWait) вместо фиксированных таймаутов.

### 2. Параллельное выполнение и гонка тестов

**Почему:** Локально тесты идут последовательно. В CI — параллельно (чтобы быстрее). Тесты влияют друг на друга: один удаляет данные, которые нужны другому.

**Диагностика:**
- Запустить тесты в CI последовательно (параллельность = 1)
- Если тесты стабилизировались — проблема в гонке
- Посмотреть CI-логи: какой тест падает, какой шел перед ним

**Решение:**
- Сделать тесты независимыми (уникальные данные через UUID)
- Или очищать данные в `@AfterEach`

### 3. Истекший токен аутентификации (401 ошибка)

**Почему:** В CI тесты идут дольше, чем время жизни токена. Например, токен живет 15 минут, а тесты идут 20 минут.

**Диагностика:**
- Посмотреть время падения первых 401 ошибок в CI
- Если 401 начинаются через 15-20 минут после старта — проблема в токене
- Локально тесты быстрее, токен не успевает протухнуть

**Решение:**
- Перезапрашивать токен перед каждым тестом (или каждые N минут)
- Добавить retry-механизм: при 401 обновить токен и повторить запрос

**Дополнительная причина (не вошла в топ-3, но частая):**
Разные версии браузера/драйвера локально и в CI. Локально Chrome 120, в CI — 115. Разное поведение.


#### Вопрос 3
Ниже приведен полный код теста (импорты опущены для краткости). Найдите в нем проблему и объясните её:

```java
public class TestDeleteUser {
    @Test
    void testDeleteUser() {
        String userId = "user-123";
        apiClient.deleteUser(userId);

        Response response = apiClient.getUserById(userId);
        assertThat(response.statusCode()).isEqualTo(404);
    }
}
```

#### Ответ 3
Название метода не соответствует тому, что на самом деле проверяется (подошло бы что то в роде getNonExistentUser).
Если это тест удаления, должен быть сценарий (Создать пользователя, Удалить его с проверкой статус кода, Проверить что
его нет в системе)
Если это тест GET на несуществующего пользователя (Создать пользователя, Удалить его, Проверить что его нет в сисеме,
Дернуть GET запрос по нему и проверить статус код 404)
